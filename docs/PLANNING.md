# 클라우드 비용 비교 서비스 기획서 (v0.1 초안)

> 상태: 초안 — 기술 스택 등 미결정 사항은 [§10](#10-미결정-사항-토론-필요)에 정리
> 작성일: 2026-07-06

---

## 1. 개요

**한 줄 정의:** 원하는 워크로드(시나리오)를 입력하면 클라우드 플랫폼별 월 예상 비용을 나란히 비교해주는 웹 서비스.

**문제 정의:**
- 각 클라우드의 가격 계산기(AWS Pricing Calculator, GCP 계산기 등)는 자기 플랫폼 안에서만 견적을 낼 수 있고, 플랫폼 간 비교를 하려면 계산기 3개를 각각 돌려서 수동으로 비교해야 한다.
- 인스턴스 이름 체계가 제각각(t3.medium vs e2-medium vs B2s)이라 "동급 스펙"을 찾는 것 자체가 진입장벽이다.
- 기존 비교 사이트(Instances.vantage.sh 등)는 단품 인스턴스 비교에 머물러 있고, "내 서비스 전체를 옮기면 얼마인가"에 답하지 못한다.

**목표:** "웹서버 2대 + DB 1대 + 스토리지 500GB + 월 트래픽 1TB" 같은 시나리오를 넣으면 3사 월 비용이 한 화면에 나오는 것. 이것 하나를 확실하게.

## 2. 타겟 사용자

| 사용자 | 상황 | 니즈 |
|---|---|---|
| 스타트업 개발자/CTO | 인프라 첫 구축 또는 이전 검토 | "우리 규모면 어디가 싸지?"에 5분 안에 답 |
| 사이드 프로젝트 개발자 | 최소 비용으로 서비스 운영 | 소규모 시나리오의 최저가 조합 |
| SI/컨설팅 엔지니어 | 고객 제안서에 견적 근거 필요 | 비교표를 공유·인용할 수 있는 출처 |

v2에서 국내 클라우드가 추가되면 "공공/금융 규제로 국내 클라우드 검토 중인 기업"이 주요 사용자로 추가된다.

## 3. 핵심 가치 제안

1. **시나리오 단위 비교** — 인스턴스 단품이 아니라 서비스 구성 전체의 월 비용을 비교. (핵심 차별화)
2. **스펙 기반 자동 매칭** — 사용자는 vCPU/RAM만 지정하면, 각 플랫폼에서 그 스펙을 만족하는 최저가 인스턴스를 자동 선택.
3. **신뢰 가능한 데이터** — 공식 가격 API에서 주기 수집, 수집 시각을 화면에 명시.

## 4. 기능 정의

### 4.1 핵심: 시나리오 빌더 + 견적 비교

**시나리오 빌더** — 리소스를 카드처럼 추가하는 UI. v1 리소스 팔레트는 4종으로 제한:

| 리소스 | 사용자 입력 | 비고 |
|---|---|---|
| 가상 머신 | vCPU, RAM(GB), 대수, (선택) OS | 스펙 최소 만족 최저가 인스턴스 자동 매칭 |
| 블록 스토리지 | 용량(GB), 타입(범용 SSD 고정) | VM에 붙는 디스크 |
| 오브젝트 스토리지 | 저장량(GB) | S3/GCS/Blob 표준 티어 |
| 아웃바운드 트래픽 | 월 전송량(GB) | 인터넷 egress, 티어 요금 계산 |

> 관리형 DB(RDS 등)는 요금 구조가 복잡해(엔진×배포방식×스토리지) v1.5로 미룬다. v1에서는 "DB도 VM으로 구성" 가이드를 제공.

**견적 비교 뷰:**
- 플랫폼별 월 총액 카드 (최저가 강조)
- 리소스별 비용 분해 테이블 — 각 플랫폼이 실제로 어떤 인스턴스를 골랐는지 표시 (예: AWS는 t3.large, GCP는 e2-standard-2)
- 통화 전환 (USD 기준 / KRW 환산)
- 공유용 URL (시나리오를 쿼리스트링/짧은 ID로 인코딩)

**시나리오 템플릿:** 빈 화면 진입장벽을 줄이기 위해 프리셋 제공 — "소규모 웹 서비스", "API 서버 + 워커", "정적 사이트 + 스토리지" 등 3~5개.

### 4.2 보조 기능 (v1 포함)

- **인스턴스 단품 비교 테이블** — 스펙 필터로 3사 인스턴스를 한 표에서 비교. 견적 엔진의 부산물로 저렴하게 만들 수 있고, SEO 유입 페이지 역할("aws gcp 인스턴스 가격 비교" 검색 유입).
- **데이터 신선도 표시** — "가격 기준: 2026-07-06 수집" 배지.

### 4.3 명시적으로 안 하는 것 (v1)

- 예약 인스턴스/약정 할인(Savings Plans, CUD) — v2
- 스팟/선점형 인스턴스 — v2
- 계정/로그인, 시나리오 저장 — v3
- Kubernetes, 서버리스, LB 등 확장 리소스 — 추후
- 리전 전체 지원 — v1은 **서울 + 버지니아(us-east-1급) 2개 리전**으로 시작

## 6. 데이터 파이프라인

### 6.1 소스 API 특성

| 플랫폼 | API | 인증 | 특이사항 |
|---|---|---|---|
| AWS | Price List Bulk API / Query API | 불필요(Bulk) | EC2 offer 파일이 리전 필터 없이 수 GB급 → 리전별 파일 사용 필수 |
| Azure | Retail Prices API (`prices.azure.com`) | 불필요 | 셋 중 가장 깔끔. OData 필터 + 페이지네이션 |
| GCP | Cloud Billing Catalog API | API 키 | SKU 설명 문자열 파싱이 필요해 셋 중 매핑 난이도 최고 |

### 6.2 구조

```
[수집기(콜렉터)] ×3사        [정규화]              [서빙]
 스케줄 실행(일 1회)   →   공통 스키마 변환   →   가격 DB/스냅샷   →   웹 앱
                            검증(급변 감지)
```

- **수집 주기:** 일 1회면 충분 (클라우드 가격은 자주 안 바뀜)
- **정규화 스키마 초안:** `(provider, service, region, sku_id, attributes{vCPU, ramGB, ...}, unit, price_usd, collected_at)`
- **검증:** 전일 대비 가격 급변(±30%↑)이나 수집 건수 급감 시 배포 보류 + 알림 — 파싱 버그가 그대로 사용자에게 노출되는 걸 방지
- **환율:** 가격은 USD로 저장, KRW 표시는 환율 API(일 1회)로 환산. "환산 기준" 명시.

### 6.3 견적 엔진 (핵심 난제)

시나리오 → 플랫폼별 비용 계산의 규칙:

1. **VM 매칭:** `요구 vCPU ≤ 인스턴스 vCPU && 요구 RAM ≤ 인스턴스 RAM` 을 만족하는 것 중 최저가. 세대 필터(너무 구세대 제외), 버스트 타입(t계열/B계열) 포함 여부는 옵션으로.
2. **egress 티어:** 3사 모두 구간별 단가 + 무료 구간이 다름 → 구간 요금표를 정규화해서 계산.
3. **공정성 원칙:** 매칭 결과(선택된 인스턴스)를 반드시 노출해서 사용자가 "비교가 공정한지" 판단할 수 있게 한다. 블랙박스 금지.

## 7. 화면 구성 (v1)

1. **홈 / 시나리오 빌더** — 좌: 리소스 카드 편집, 우: 실시간 3사 견적 요약. 템플릿 선택 진입점.
2. **비교 결과 상세** — 리소스×플랫폼 비용 분해 테이블, 선택된 인스턴스 명세, 공유 버튼. (빌더와 같은 페이지의 확장 영역이어도 됨)
3. **인스턴스 비교표** — 필터 가능한 단품 비교 테이블. SEO 랜딩.
4. **데이터 안내** — 수집 방식/시각/한계 고지.

## 8. 기술 스택

### 결정됨 (2026-07-06)
- 데이터 수집: 공식 API 배치 (§6)
- 웹: **Next.js 풀스택 (App Router)**, Vercel 배포 — 백엔드가 가볍고, 비교표 페이지의 SEO가 중요해서
- 가격 데이터 저장소: **DB 없음, 레포에 커밋되는 JSON 스냅샷** — v1 데이터 규모(3사×2리전×4리소스)가 몇 MB 수준이라 DB는 오버엔지니어링
- 수집기: **GitHub Actions cron**(일 1회)에서 수집 스크립트 실행 → 정규화 JSON을 레포에 커밋 → 커밋이 Vercel 자동 재배포를 트리거

```
GitHub Actions (일 1회 cron)
  └─ 수집 스크립트 (AWS/Azure/GCP API 호출)
      └─ 정규화 + 검증 → data/*.json 커밋 & push
          └─ Vercel이 push 감지 → 자동 빌드·배포 (배포 = 데이터 갱신)
```

부수 효과: 가격 데이터가 git 히스토리에 쌓이므로 **가격 변동 이력을 공짜로 확보** — 추후 "가격 추이" 기능의 기반이 된다.

## 11. 성공 지표 (v1)

- 시나리오 입력 → 견적 확인까지 이탈 없이 도달하는 비율
- 공유 URL 생성 수 (제안서/커뮤니티 인용 = 핵심 확산 경로)
- 인스턴스 비교표 검색 유입 수
